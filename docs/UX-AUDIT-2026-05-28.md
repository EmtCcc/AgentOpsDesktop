# AgentOps Desktop — Visual & UX Audit

**Date**: 2026-05-28
**Scope**: Full renderer codebase (`index.html`, `app.js`, all CSS files)
**Phase**: Foundation v0.1 — vanilla HTML/CSS/JS, React SPA planned

---

## 1. Design Patterns — Strengths

- **Consistent token system**: `tokens.css` defines a clean, well-organized token vocabulary (colors, spacing, typography, shadows, motion, z-index). All CSS references tokens, no hard-coded values in components.
- **Dark mode via media query**: Automatic `prefers-color-scheme: dark` support with full token override — no JS toggle needed.
- **Component modularity**: CSS split into `tokens → base → layout → components → pages` — clean cascade, low specificity conflicts.
- **Reduced motion**: `prefers-reduced-motion` is respected globally in `base.css:171-179`.
- **Focus-visible**: All interactive elements get `outline: 2px solid` on `:focus-visible` — keyboard navigation works.
- **Semantic HTML**: `header`, `nav`, `main`, `footer`, `role="region"`, `role="log"`, `aria-label`, `aria-live="polite"` are used on dynamic regions.
- **Status dot system**: 4-state color coding (running/idle/error/spawning) is consistent across dots, badges, and footer.

## 2. Design Patterns — Issues

### 2.1 Modal implementation bypasses design system
**Files**: `app.js:202-237`, `app.js:371-403`

Both the Agent and Task modals use **inline styles** instead of the `.modal-overlay` / `.modal` classes defined in `pages.css:714-778`. The modals are built with raw `style="display:none; position:fixed; ..."` instead of reusing the component system.

**Impact**: Inconsistent modal appearance, harder to maintain, missing backdrop blur defined in `.modal-overlay`.

### 2.2 Duplicate settings icon in header + sidebar
**Files**: `index.html:47-49`, `index.html:118-123`

The settings gear icon appears both in the header actions and in the sidebar footer. Both navigate to the same page. This is redundant and wastes header space.

### 2.3 User menu in header is non-functional
**File**: `index.html:53-59`

The header user menu (`#user-menu`) has hover styling but no click handler or dropdown. It displays static "AgentOps / Admin" text. Either wire it up or remove it to avoid user confusion.

### 2.4 Notification badge always visible but non-functional
**File**: `index.html:42-45`

The notification button shows a red badge dot unconditionally (no logic to hide when 0 notifications). No click handler exists. Should be hidden or wired.

### 2.5 Landing page referenced but not rendered
**Files**: `index.html:73-78`, `app.js:43-58`

Sidebar has a "Home" link (`data-page="landing"`), but `app.js` has no `renderLanding` function — it falls through to "Page not found". The landing page CSS (`pages.css:537-627`) exists but is unused.

### 2.6 Workflows page referenced but not rendered
**File**: `index.html:107-113`

Sidebar has "Workflows" but no `renderWorkflows` in `app.js`. Falls to "Page not found".

## 3. Content Quality

### 3.1 Dashboard stat cards lack context
**File**: `app.js:64-101`

Stats show raw numbers (0 Agents, 0 Tasks, 0 Running, 0 Errors) without trend indicators, timestamps, or links to drill down. The `stat-card__change` CSS class exists (`components.css:239-254`) but is never used.

### 3.2 Task cards show agent ID, not agent name
**File**: `app.js:434`

Tasks display `t.assigneeAgentId` (a UUID) instead of resolving to the agent name. Users see `unassigned` or a raw UUID — not human-readable.

### 3.3 Empty states are minimal
The empty states for agents, tasks, and logs are functional but generic. The agent empty state has a CTA button ("Add agent") — good. Task board and log viewer empty states lack actionable CTAs.

### 3.4 Settings page has no persistence
**File**: `app.js:498-566`

All settings inputs (max parallel agents, task timeout, log retention) are hardcoded HTML with no state binding. Changes are lost on navigation. No save button.

### 3.5 Error handling uses toast but no inline validation
**File**: `app.js:700-719`

Agent creation silently ignores empty name (`if (!name) return`) with no visual feedback. The toast system works but is only used for IPC failures, not form validation.

## 4. UX Observations

### 4.1 Sidebar collapse doesn't preserve state
**File**: `app.js:742-749`

Sidebar toggle is in-memory only. On app restart, it resets to expanded. Should persist to `localStorage` or the JSON store.

### 4.2 No loading states
`app.js:141-153`, `app.js:244-269`

Data fetches (`loadStats`, `loadAgents`, `loadTasks`) show nothing while loading. The skeleton CSS exists (`components.css:596-633`) but is never used. Users see blank content until IPC responds.

### 4.3 No confirmation for destructive actions
**File**: `app.js:315-323`

Agent deletion (`data-action="delete"`) calls `window.agentOps.agents.delete(id)` immediately with no confirmation dialog. Accidental clicks destroy data.

### 4.4 Task board has no drag-and-drop
**File**: `app.js:329-408`

The kanban board renders 4 static columns but tasks can't be moved between them. Status changes require editing — not discoverable.

### 4.5 Search bar does nothing
**File**: `index.html:35-38`

The global search input (`#global-search`) has a `⌘K` placeholder but no handler. Typing does nothing. Should either be wired or hidden.

### 4.6 Keyboard shortcut `⌘K` not implemented
The search placeholder references `⌘K` but no keydown listener exists for this shortcut.

### 4.7 Log viewer auto-scroll threshold is fragile
**File**: `app.js:657`

Auto-scroll triggers when within 80px of bottom. If the user scrolls up to read, new entries still auto-scroll down — no way to "pause" and read historical logs without losing position.

### 4.8 Log level filter applies client-side after fetching all logs
**File**: `app.js:629`

The level filter only filters the already-fetched 500 entries. If there are 10,000 error logs, fetching 500 and filtering may miss them. Should pass level to the IPC query.

## 5. Accessibility Audit

### 5.1 Passes
- `lang="en"` on `<html>`
- `aria-label` on regions (dashboard stats, task board, agent list, log viewer)
- `aria-live="polite"` on dynamic content regions
- `role="log"` on activity feed and log viewer
- `role="region"` on dashboard stats
- `:focus-visible` outline on all interactive elements
- `prefers-reduced-motion` disables animations
- Color palette has sufficient contrast (WCAG AA for text on backgrounds)

### 5.2 Fails

| Issue | Location | Severity |
|-------|----------|----------|
| **SVG icons lack `aria-hidden="true"`** | All inline SVGs in `index.html` and `app.js` | Medium — screen readers announce raw SVG content |
| **Status dots have no text alternative** | `app.js:254`, `components.css:258-268` | High — color-only status indication fails for colorblind users |
| **Modal overlay lacks `role="dialog"` and `aria-modal`** | `app.js:202`, `app.js:371` | High — screen readers don't trap focus in modals |
| **Modal overlay lacks `Escape` key handler** | `app.js:272-324` | Medium — keyboard users can't dismiss modals |
| **Form inputs in modals lack `for`/`id` association** | `app.js:209-230` | Medium — `<label>` uses visual proximity, not `for` attribute |
| **Sidebar nav items are `<a>` without `href`** | `index.html:67-112` | Low — semantically should be `<button>` since they trigger JS navigation |
| **No skip-to-content link** | `index.html` | Low — keyboard users must tab through entire sidebar to reach main content |
| **Dashboard stat values lack `aria-label`** | `app.js:76-99` | Low — screen readers read "0" without context ("0 agents") |
| **Toast container created dynamically without announcement** | `app.js:700-708` | Low — `role="alert"` is set, but container is created on first error, which may miss the announcement |

### 5.3 Contrast Ratios (estimated)

| Pair | Light Mode | Dark Mode | WCAG AA (4.5:1) |
|------|-----------|-----------|-----------------|
| `--color-text-secondary` (#6B7280) on `--color-bg-primary` (#FFF) | ~5.9:1 | Pass |
| `--color-text-tertiary` (#9CA3AF) on `--color-bg-primary` (#FFF) | ~3.9:1 | **Fail** for small text |
| `--color-text-tertiary` (#9CA3AF) on `--color-bg-secondary` (#F9FAFB) | ~3.7:1 | **Fail** for small text |
| `--color-primary` (#6366F1) on `--color-bg-primary` (#FFF) | ~4.6:1 | Pass (borderline) |
| `--color-text-tertiary` in dark (#9CA3AF) on `--color-bg-primary` (#111827) | ~5.4:1 | Pass |

**Issue**: `--color-text-tertiary` fails WCAG AA for small text in light mode. Used for timestamps, placeholders, captions.

## 6. Visual Consistency Issues

### 6.1 Inline styles throughout `app.js`
**Lines**: `app.js:110-111`, `app.js:202-237`, `app.js:371-403`, `app.js:425`, `app.js:525`, `app.js:582`, `app.js:597`

Approximately 30+ inline `style=` attributes in the JS templates. These bypass the design system tokens and make the UI harder to maintain. Examples:
- `style="padding: var(--space-8) 0;"` — OK (uses tokens but should be a class)
- `style="display:none; position:fixed; inset:0; background:rgba(0,0,0,0.5);"` — should be `.modal-overlay`
- `style="width: 440px; max-width: 90vw;"` — should be a modal size class

### 6.2 Mixed use of `.card` and `.stat-card` patterns
Dashboard uses `.card` with inner `.stat` layout (`app.js:73-99`) while `components.css` defines `.stat-card` with its own icon/value/label structure. The dashboard doesn't use `.stat-card`.

### 6.3 Log filter dropdowns use inline sizing
**File**: `app.js:582-591`

Log filter `<select>` elements use inline `style="height:28px; font-size:var(--text-xs);"` instead of the `.logs-filter` class defined in `pages.css:526-534` (which sets `height: 36px`).

## 7. Responsive Design

### 7.1 Passes
- Sidebar collapses to icon-only at `max-width: 1024px`
- Grid layouts collapse from 4→2 columns at `max-width: 960px`
- Modal has `max-width: 90vw` constraint

### 7.2 Fails
- **Dashboard stats grid**: `grid-template-columns: repeat(4, 1fr)` at all widths — at `<768px`, 4 stat cards squeeze too tight. No responsive breakpoint.
- **Task board**: `grid-template-columns: repeat(4, 1fr)` — at `<1024px`, 4 columns become unreadable. Should collapse to 2 or 1.
- **Log viewer height**: `calc(100vh - header - footer - 200px)` — hardcoded offset doesn't account for the page header, which varies.

---

## 8. Performance Observations

- **No virtualization**: Log viewer renders up to 500 DOM nodes. For high-volume agents, this will cause jank. Consider windowed rendering.
- **XSS protection**: `escapeHtml()` is used consistently on user content — good.
- **CSP**: `Content-Security-Policy` header is set in `index.html:6` — good baseline.
- **No code splitting**: Single `app.js` bundles all page renderers. Fine for v0.1, will need splitting when React lands.
